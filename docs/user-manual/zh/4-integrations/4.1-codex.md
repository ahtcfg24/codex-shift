# 4.1 Codex 集成

`codex-shift` 可以作为 Codex 的 Responses provider。这样 Codex 只需要连接本地代理，实际请求由代理路由到 DeepSeek、MiMo、内部 Responses 网关等上游。

## 集成目标

完成集成后，Codex 可以：

- 通过本地 `base_url` 调用 `POST /v1/responses`。
- 使用 `model_map` 将 Codex 中的模型名映射到上游模型名。
- 在会话启动时从 `/models` 获取真实 `context_window` 与 `max_context_window`。
- 不直接持有上游真实 API Key。

## 关键限制：272k fallback 上限

当 Codex 未能从 `/models` 拿到某模型元数据时，会对未知模型套用 fallback。该 fallback 的 `context_window` 与 `max_context_window` 均为 `272000`。

因此，仅在 Codex `config.toml` 中设置：

```toml
model_context_window = 1000000
```

并不能真正得到 1M 上下文，因为配置会被 fallback 的 `max_context_window` 截断。要突破 272k，需要让 Codex 请求本代理的 `/models`，并由代理返回更大的 `max_context_window`。

## 触发 `/models` 的方式

Codex 只有在满足触发条件时才会请求：

```text
GET {base_url}/models
```

已验证的触发方式是为 provider 配置 command auth。仅配置 `base_url` 与普通环境变量 Key 不会触发 `/models`。

## 推荐配置

在 Codex `config.toml` 中配置 provider：

```toml
model = "deepseek-v4-pro"
model_provider = "box"

[model_providers.box]
name = "codex_shift"
base_url = "http://127.0.0.1:8080/v1"
wire_api = "responses"

[model_providers.box.auth]
command = "printf"
args = ["%s", "dummy-token"]
timeout_ms = 5000
refresh_interval_ms = 0
cwd = "."
```

说明：

- `base_url` 指向本代理的 `/v1` 前缀。
- `wire_api = "responses"` 让 Codex 发送 Responses 请求。
- `auth.command` 输出任意 token 即可，因为上游真实 Key 由本代理按 provider 配置注入。
- 配置 `auth` 后，Codex 会在会话启动时真正请求 `/models`。

## 代理配置示例

`config.yaml` 中声明模型上下文窗口：

```yaml
model_map:
  gpt-5.5: deepseek-v4-pro

providers:
  - name: deepseek
    enabled: true
    weight: 1
    outbound: chat_completions
    base_url: https://api.deepseek.com
    api_key_env: DEEPSEEK_API_KEY
    models:
      - name: deepseek-v4-pro
        context_window: 1000000
        max_context_window: 1000000
```

如果 Codex 中直接使用 `model = "deepseek-v4-pro"`，无需 `model_map` 也可以路由。`model_map` 适合希望在客户端保留通用模型名的场景。

## 验证 `/models` 是否被调用

启动代理后运行 Codex，会话启动时代理日志中应出现类似记录：

```text
入站 GET /v1/models?client_version=...
```

也可以手动检查：

```bash
curl http://127.0.0.1:8080/v1/models
```

确认响应中包含目标模型，并且 `context_window` / `max_context_window` 是你在 `config.yaml` 中配置的值。

## Codex 模型缓存

Codex 会缓存 `/models` 结果，缓存文件通常位于：

```text
~/.codex/models_cache.json
```

如果你修改了模型上下文窗口但 Codex 仍使用旧值，可以删除缓存后重启 Codex：

```bash
rm ~/.codex/models_cache.json
```

## 关于 base_instructions

Codex 拉取 `/models` 后，会使用返回的 `ModelInfo` 整体覆盖该 slug 的元数据，包括 `base_instructions` 与工具能力字段。

`codex-shift` 的 `/models` 条目默认基于内置 `gpt-5.5` 模板克隆，因此已经带有完整的 Codex `ModelInfo` 字段。你可以通过以下方式覆盖系统提示词：

- 模型级 `base_instructions`。
- 全局 `defaults.base_instructions`。
- Codex `config.toml` 中的 `base_instructions`，优先级最高。
- `defaults.model_template` 指向自定义 `ModelInfo` 模板文件。

## 常见问题

### 配置了 1M 上下文但仍只有 272k

检查 Codex 是否真的请求了 `/models`。如果没有请求，通常是 provider 未配置 command auth。

### `/models` 返回为空

检查目标模型是否配置了 `context_window`，或全局是否设置了 `defaults.context_window`。未配置上下文窗口的模型不会出现在 `/models` 中。

### Codex 仍使用旧模型信息

删除 `~/.codex/models_cache.json` 后重启 Codex。

### 上游返回 401

Codex 侧的 dummy token 不用于上游鉴权。请检查本项目 `.env` 与 `config.yaml` 中 provider 的 `api_key_env` 是否匹配。
