# 3.1 端点说明

`codex-shift` 对外暴露 Responses 入站、模型目录、健康检查和本地控制台端点。默认服务地址由 `config.yaml` 中的 `server.host` 与 `server.port` 决定。

本文示例使用：

```text
http://127.0.0.1:8080
```

## `POST /v1/responses`

兼容 OpenAI Responses 风格的主入站端点。`POST /responses` 也可用，便于兼容不同客户端的 `base_url` 拼接方式。

### 请求示例

```json
{
  "model": "gpt-5.5",
  "input": [
    {"role": "user", "content": "Hello!"}
  ],
  "stream": true
}
```

### 支持参数

| 参数 | 说明 |
|------|------|
| `model` | 入站模型名，经 `model_map` 映射后路由到 provider |
| `input` | 字符串或输入项列表 |
| `stream` | 是否流式返回 |
| `tools` | function 定义，Chat 出站支持 namespace 展平 |
| `tool_choice` | 工具调用策略 |
| `reasoning.effort` | 推理力度，Chat 出站映射为 `reasoning_effort` |
| `max_output_tokens` | 最大输出 token 数 |
| `temperature` / `top_p` | 采样参数 |
| `text.format` | Chat 出站映射为 `response_format` |
| `parallel_tool_calls` | Chat 出站原样透传 |

### 非流式请求

```bash
curl http://127.0.0.1:8080/v1/responses \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "gpt-5.5",
    "input": [{"role": "user", "content": "Hello!"}],
    "stream": false
  }'
```

Chat 出站时，代理会将上游 Chat Completions 响应转换为 Responses 风格响应。Responses 出站时，代理原样回送上游 Responses 响应。

### 流式请求

```bash
curl -N http://127.0.0.1:8080/v1/responses \
  -H 'Content-Type: application/json' \
  -d '{
    "model": "gpt-5.5",
    "input": [{"role": "user", "content": "Hello!"}],
    "stream": true
  }'
```

Chat 出站时，上游 SSE 会转换为 Responses 事件序列，例如 `output_text.delta` 与 `function_call_arguments.delta`。Responses 透传时，上游 SSE 文本块原样转发。

### 错误格式

代理会尽量把错误转换成统一结构：

```json
{
  "error": {
    "message": "上游请求超时",
    "type": "timeout_error"
  }
}
```

常见错误类型：

| 类型 | 场景 |
|------|------|
| `invalid_request_error` | 请求体不是合法 JSON 或结构不符合要求 |
| `model_not_found` | 无法为模型找到匹配 provider |
| `conversion_error` | 请求转换失败 |
| `timeout_error` | 上游请求超时 |
| `upstream_error` | 上游请求失败或返回错误 |

## `GET /models`

`GET /models` 与 `GET /v1/models` 返回 Codex 兼容的模型目录。

### 响应示例

```jsonc
{
  "models": [
    {
      "slug": "deepseek-v4-pro",
      "display_name": "deepseek-v4-pro",
      "context_window": 1000000,
      "max_context_window": 1000000,
      "effective_context_window_percent": 95,
      "base_instructions": "You are Codex, a coding agent ..."
      // 其余字段来自内置 ModelInfo 模板
    }
  ]
}
```

返回说明：

- 该端点返回 Codex `ModelInfo` 结构，不是 OpenAI 标准 `{"object":"list","data":[...]}`。
- 只收录已配置上下文窗口的模型。
- 条目基于内置 `codex_shift/data/model_template.json` 克隆。
- 响应带稳定 `ETag`，支持 `If-None-Match` 条件请求，命中时返回 `304`。

### 何时出现模型

模型会出现在 `/models` 中的条件：

- provider 已启用。
- 模型对象配置了 `context_window`，或全局 `defaults.context_window` 已设置。
- 模型经 `model_map` 形成的别名也会纳入模型目录。

## `GET /health`

健康检查端点，服务就绪时返回 `200`。

```bash
curl http://127.0.0.1:8080/health
```

响应示例：

```json
{
  "status": "ok",
  "providers": [
    {
      "name": "deepseek",
      "enabled": true,
      "weight": 1.0,
      "outbound": "chat_completions",
      "upstream": "https://api.deepseek.com/v1/chat/completions",
      "models": ["deepseek-v4-pro"],
      "catch_all": false
    }
  ]
}
```

该端点适合用于启动后探活、脚本验证和本地运行状态检查。

## `GET /admin`

本地控制台页面，与 API 服务运行在同一个端口。

```text
http://127.0.0.1:8080/admin
```

控制台支持：

- 查看所有 provider。
- 修改 provider 的 `enabled`。
- 修改 provider 的 `weight`。
- 保存后写回 `config.yaml` 并热加载。

控制台只更新 provider 的启停与权重，不负责编辑全部配置字段。

## `GET /admin/api/config`

控制台使用的配置读取接口，返回 provider 控制项摘要。

```json
{
  "providers": [
    {
      "name": "deepseek",
      "enabled": true,
      "weight": 1.0,
      "outbound": "chat_completions",
      "base_url": "https://api.deepseek.com",
      "path": "/v1/chat/completions",
      "models": ["deepseek-v4-pro"],
      "catch_all": false
    }
  ]
}
```

## `POST /admin/api/config`

控制台使用的保存接口。请求体必须包含 provider 的 `name`、`enabled` 与 `weight`。

```json
{
  "providers": [
    {"name": "deepseek", "enabled": true, "weight": 2},
    {"name": "mimo", "enabled": false, "weight": 1}
  ]
}
```

保存流程：

1. 读取当前 `config.yaml`。
2. 只更新匹配 provider 的 `enabled` 与 `weight`。
3. 写入临时配置并调用配置加载逻辑验证。
4. 验证通过后原子替换正式配置。
5. 进程内热加载新配置。

如果配置验证失败，正式配置不会被替换。
