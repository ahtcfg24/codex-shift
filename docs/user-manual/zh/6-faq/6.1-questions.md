# 6.1 FAQ

## codex-shift 支持哪些入站协议？

当前入站协议是 OpenAI Responses 风格，请求端点为：

- `POST /v1/responses`
- `POST /responses`

项目不提供 Anthropic Messages 入站端点。

## 支持哪些出站协议？

provider 的 `outbound` 支持两个值：

| 值 | 说明 |
|----|------|
| `chat_completions` | 上游为 OpenAI Chat Completions 兼容接口，代理做双向转换 |
| `responses` | 上游为 OpenAI Responses 接口，代理做全透传 |

不支持 `outbound: messages`。

## 为什么不支持 Anthropic Messages？

OpenAI Responses 与 Anthropic Messages 在会话模型、内容块、工具调用、reasoning/thinking、流式事件、错误语义以及缓存机制上差异较大。

尤其是 Anthropic Messages 的 prompt caching / `cache_control` 与 OpenAI Responses 的上下文和状态管理难以可靠兼容。做近似转换容易产生隐性语义偏差、缓存失效或上游 `400`，因此项目只维护 Responses 与 Chat Completions，以及 Responses 全透传。

## `config.yaml` 和 `.env` 应该提交吗？

不应该。两者都可能包含真实 API Key 或内部服务地址，已在 `.gitignore` 中排除。

推荐提交和共享的是：

- `config.example.yaml`
- `.env.example`
- 文档中的脱敏示例

## Provider 的 `api_key` 和 `api_key_env` 谁优先？

`api_key` 优先。如果 `api_key` 非空，代理会直接使用它；否则读取 `api_key_env` 指定的环境变量。

更推荐使用 `api_key_env`，并把真实 Key 写入本地 `.env` 或 shell 环境。

## `/models` 为什么没有我的模型？

常见原因：

- provider 被设置为 `enabled: false`。
- 模型没有配置 `context_window`，且全局没有设置 `defaults.context_window`。
- provider 的 `models[].name` 没有声明客户端使用的入站模型名。
- 服务未热加载最新配置，可查看 `/health` 或重启服务确认。

## Codex 为什么还是 272k 上下文？

通常是 Codex 没有请求本代理的 `/models`。请检查 Codex provider 是否配置了 command auth。

推荐配置见 [Codex 集成](../4-integrations/4.1-codex.md)。

## 修改 `config.yaml` 后需要重启吗？

取决于修改方式：

- 通过 `/admin` 修改 provider、模型列表、模型上下文窗口、`enabled` / `weight` 后，会自动热加载。
- 手动编辑 `config.yaml` 后，建议运行 `./start.sh restart`。

## 控制台能编辑所有配置吗？

不能覆盖所有高级字段，但可以完成常用 provider 管理。

`/admin` 支持添加/删除 provider、修改上游地址、协议类型、API Key 环境变量、模型列表、`mapped_model` 和模型级 `context_window`。如果需要维护 `defaults`、高级模型元数据或全局 `options`，请编辑 `config.yaml`。

## 启动脚本为什么改了我的端口？

`start.sh` 会在启动时检测 `config.yaml` 中的端口是否可绑定。如果端口已被占用，会从 `10000` 起寻找可用端口，并把新端口写回 `config.yaml`。

这样可以避免服务启动失败。若需要固定端口，请先释放原端口，再手动改回配置。

## 可以暴露到公网吗？

不建议直接暴露。项目本身是本地代理，没有内置用户认证。

如果必须让其他机器访问，请至少做到：

- 使用反向代理或网关加认证。
- 只允许可信 IP 访问。
- 不开启 debug 日志。
- 定期检查日志与配置文件权限。

## `web_search` 为什么没有生效？

只有 Chat 出站且 provider 配置支持时才会透传 `web_search`。请检查：

- provider 是否设置 `web_search: true`。
- 上游账号或套餐是否开通联网搜索能力。
- 上游是否需要额外的 `webSearchEnabled: true`，如果需要，请设置 `web_search_enabled: true`。

Responses 全透传模式不处理工具语义，工具是否生效完全由上游决定。

## 已知限制

- `text.format` 仅在 Chat 出站时映射为 `response_format`；Responses 透传时原样交给上游。
- Responses 内置工具中，Chat 出站仅 `web_search` 可在 provider 开启支持时透传。
- `file_search`、`computer_use` 等 Responses 内置工具不会被转换为 Chat 工具。
- 代理层不维护 `previous_response_id`、`conversation`、`store` 等有状态能力。
- `reasoning` 输入项不重放，加密内容无法安全还原。

## 如何贡献文档？

改动用户可见行为时，请同时更新：

- 根目录 `README.md` 中的入口说明或链接。
- `docs/user-manual/zh/` 下对应章节。
- `config.example.yaml` 中相关注释。

提交前建议检查 Markdown 链接是否仍指向真实文件。
