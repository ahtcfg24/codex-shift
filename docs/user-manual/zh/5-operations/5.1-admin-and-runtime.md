# 5.1 控制台与运行状态

## 本地控制台

启动服务后访问：

```text
http://127.0.0.1:8080/admin
```

如果 `start.sh` 因端口占用自动选择了新端口，请使用启动输出中的实际端口。

控制台用于运行时调整 provider 控制项：

| 控制项 | 说明 |
|--------|------|
| `enabled` | 是否启用 provider。关闭后不参与路由和 `/models` |
| `weight` | 同一模型多 provider 候选时的路由权重 |

保存后，服务会写回 `config.yaml` 并热加载进程内配置。后续请求会使用新配置，不需要重启服务。

## 控制台保存机制

控制台保存时只更新 provider 的 `enabled` 与 `weight` 字段，不会修改 `base_url`、`api_key_env`、`models` 等其他配置。

保存流程：

1. 读取当前 `config.yaml`。
2. 根据 provider `name` 匹配更新项。
3. 写入临时 YAML 文件。
4. 使用正式配置加载逻辑验证临时文件。
5. 验证通过后原子替换 `config.yaml`。
6. 热加载到进程内。

如果验证失败，正式配置不会被替换，控制台会收到错误响应。

## 查看运行状态

### 使用启动脚本

```bash
./start.sh status
```

运行中时会输出后台进程 PID；未运行时返回非零状态码。

### 使用健康检查

```bash
curl http://127.0.0.1:8080/health
```

健康检查会返回当前启用 provider 的摘要，包括出站协议、上游地址、模型列表和权重。

## 日志

`start.sh` 默认把服务日志追加到：

```text
.codex-shift.log
```

查看实时日志：

```bash
tail -f .codex-shift.log
```

日志中会记录：

- 入站请求方法、完整 path 和来源地址。
- 请求 ID，用于串联同一次请求的多条日志。
- 模型映射与 provider 路由结果。
- 上游请求状态。
- 流式转发完成情况。
- 转换错误、上游错误、非法 JSON 和超时。

debug 模式会额外记录请求头、入站请求体、出站请求体和上游响应体。生产或共享环境中请谨慎开启 debug 日志，避免敏感内容进入日志文件。

## 端口占用处理

`start.sh` 启动时会读取 `config.yaml` 中的 `server.host` 与 `server.port`。如果目标端口已被占用，脚本会：

1. 从 `10000` 起寻找可用端口。
2. 找到后写回 `config.yaml` 的 `server.port`。
3. 使用新端口启动服务。
4. 在启动输出中显示最终监听地址。

如果你希望固定端口，请先停止占用该端口的进程，再把 `config.yaml` 改回目标端口。

## 停止与重启

```bash
./start.sh stop
./start.sh restart
```

停止时，脚本会优先读取 `.codex-shift.pid`，向对应进程发送终止信号，并清理 PID 文件。随后会按命令行特征清理可能残留的历史实例。

## 常用排查步骤

### 服务未启动

1. 查看 `.codex-shift.log`。
2. 确认 Python 版本满足要求。
3. 确认 `requirements.txt` 依赖安装成功。
4. 运行 `./start.sh --log-level debug` 获取更多日志。

### 上游返回 401

1. 检查 `.env` 中是否填写了真实 API Key。
2. 检查 provider 的 `api_key_env` 是否与 `.env` 变量名一致。
3. 确认当前启动方式会加载项目 `.env`。

### 模型无法路由

1. 查看错误中的“映射后”模型名。
2. 检查 `model_map` 是否符合预期。
3. 确认某个启用 provider 的 `models` 中声明了映射后的模型名。
4. 若使用控制台关闭过 provider，确认它仍处于启用状态。

### 流式请求失败

1. 检查上游是否支持流式响应。
2. 查看日志中是否出现上游错误状态。
3. 如果是 Chat 出站，确认 provider 是否兼容 OpenAI Chat Completions SSE 格式。
4. 如果是 Responses 透传，确认上游返回的是 Responses SSE 格式。

## 安全建议

- 本地使用建议监听 `127.0.0.1`。
- 不要把 `.env`、`config.yaml` 或日志文件提交到仓库。
- debug 日志可能包含请求内容和上游响应，排查结束后应切回普通日志级别。
- 如果将服务暴露给其他机器，请在外层加入认证和访问控制。
