# 1.1 软件介绍

## 什么是 codex-shift

`codex-shift` 是一个面向 Codex 的本地 Responses 协议适配层。它让 Codex 通过一个稳定的本地入口调用多个模型后端，并在代理层处理协议转换、模型映射、API Key 注入和 provider 路由。

服务运行在本机，通过 FastAPI 暴露 HTTP 端点。Codex 只需要配置一个 `base_url`，具体的模型映射、provider 选择、API Key 注入和协议转换由适配层完成。

## 解决什么问题

在本地使用 Codex 接入第三方模型或内部网关时，经常会遇到这些问题：

- **协议不一致**：有的上游支持 Chat Completions，有的上游支持 Responses，Codex 侧不应为每个上游维护不同接入方式。
- **模型名称不统一**：Codex 使用的模型名和上游真实模型名可能不同，需要稳定映射。
- **上下文窗口不准确**：Codex 对未知模型存在 fallback 上限，需要通过 `/models` 获取真实 `context_window`。
- **多 provider 切换麻烦**：同一模型可能由多个 provider 提供，需要按权重或开关控制流量。
- **密钥不应暴露给客户端**：API Key 可以只保存在本地代理环境或配置中，由代理注入到上游请求。

`codex-shift` 将这些差异收敛到本地配置文件中，让 Codex 只面对统一入口。

## 核心功能

### Responses 入站

- 支持 `POST /v1/responses` 与 `POST /responses`。
- 请求体使用 OpenAI Responses 风格。
- 支持流式与非流式响应。

### Provider 路由

- 一个配置文件中可声明多个 provider。
- 入站 `model` 可通过 `model_map` 映射为上游真实模型名。
- 同一模型存在多个 provider 候选时，按 `weight` 加权选择。
- `enabled: false` 的 provider 保留在配置中，但不参与路由。

### 协议转换

| 出站协议 | 行为 |
|----------|------|
| `chat_completions` | 将 Responses 请求转换为 Chat Completions，再将上游响应转换回 Responses |
| `responses` | 上游本身就是 Responses 协议时全透传，仅注入 API Key 并保留模型目录能力 |

### 模型目录

`GET /models` 与 `GET /v1/models` 返回 Codex 兼容的模型列表。模型条目基于内置 `ModelInfo` 模板克隆，只覆盖模型名、上下文窗口等配置字段，方便 Codex 读取真实上下文窗口。

### 本地控制台

`GET /admin` 提供本地控制台，可以查看 provider，修改 `enabled` 与 `weight`，保存后热加载到进程内，不需要重启服务。

## 工作流程

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Box as codex-shift
    participant Provider as 上游 Provider

    Client->>Box: POST /v1/responses
    Box->>Box: 解析 model / model_map
    Box->>Box: 按 provider enabled + weight 路由
    Box->>Provider: 转换或透传请求
    Provider-->>Box: 返回 Chat 或 Responses 响应
    Box-->>Client: 回送 Responses 响应
```

## 适合人群

- 需要在 Codex 中接入自定义模型或第三方 provider 的用户。
- 需要把多个模型后端统一成 Codex Responses 入口的开发者。
- 需要在本地集中管理 API Key、模型映射和 provider 权重的团队或个人。

## 不适合的场景

- 需要 Anthropic Messages 作为入站或出站协议的场景。
- 需要代理层维护有状态会话、文件搜索、电脑控制等完整 Responses 扩展语义的场景。
- 需要把代理暴露到公网且缺少额外认证层的场景。
