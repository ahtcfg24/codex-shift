# 2.1 Provider 配置

## 配置文件位置

主配置文件为项目根目录的 `config.yaml`。首次启动时，`start.sh` 会在缺少该文件时从 `config.example.yaml` 复制。

推荐做法：

```bash
cp config.example.yaml config.yaml
cp .env.example .env
```

真实 API Key 建议写在 `.env` 中，并通过 `api_key_env` 引用。

## 顶层结构

```yaml
server:        # 本地监听地址与端口
providers:     # 多 provider 列表
defaults:      # /models 目录的全局默认值
options:       # 全局转换选项
```

## server

```yaml
server:
  host: 127.0.0.1
  port: 8080
```

| 字段 | 说明 |
|------|------|
| `host` | 服务监听地址。推荐本地使用 `127.0.0.1` |
| `port` | 服务监听端口。启动脚本检测到占用时会自动选择新端口并写回配置 |

如果监听 `0.0.0.0`，请确保外层网络和认证策略足够安全。

## 模型映射位置

推荐把模型映射写在 provider 内部的 `models[].mapped_model` 中。

```yaml
providers:
  - name: deepseek
    models:
      - name: gpt-5.5
        mapped_model: deepseek-v4-pro
        context_window: 1000000
```

其中：

- `name` 是入站模型名，也是路由表和 `/models` 目录展示的模型名。
- `mapped_model` 是当前 provider 实际收到的上游模型名，省略时默认与 `name` 相同。

顶层 `model_map` 仅作为旧配置兼容路径保留，不建议新配置继续使用。

## providers

每个 provider 代表一个可调用的上游服务。

```yaml
providers:
  - name: deepseek
    enabled: true
    weight: 1
    outbound: chat_completions
    base_url: https://api.deepseek.com
    api_key_env: DEEPSEEK_API_KEY
    timeout: 300
    models:
      - name: gpt-5.5
        mapped_model: deepseek-v4-pro
        context_window: 1000000
      - deepseek-v4-flash
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 条件 | Provider 名称，需唯一。列表写法未填时会自动生成 `provider_<index>` |
| `enabled` | 否 | 是否启用，默认 `true`。关闭后不参与路由和 `/models` |
| `weight` | 否 | 同一模型多 provider 候选时的加权路由权重，默认 `1` |
| `outbound` | 是 | 出站协议：`chat_completions` 或 `responses` |
| `base_url` | 是 | 上游基础地址，不含默认路径 |
| `api_key` | 条件 | 直接写入 API Key。优先级高于 `api_key_env` |
| `api_key_env` | 条件 | 从环境变量读取 API Key |
| `models` | 条件 | 该 provider 支持的模型列表 |
| `path` | 否 | 覆盖默认出站路径 |
| `timeout` | 否 | 上游超时秒数，默认 `300` |
| `passthrough_unknown` | 否 | Chat 出站时是否透传未识别顶层字段 |
| `web_search` | 否 | 是否允许 Chat 出站透传 Responses `web_search` 工具 |
| `web_search_enabled` | 否 | 携带 `web_search` 时是否附加 `webSearchEnabled: true` |

## 出站协议

### chat_completions

```yaml
outbound: chat_completions
base_url: https://api.deepseek.com
# 默认 path: /v1/chat/completions
```

代理会把入站 Responses 请求转换为 Chat Completions 请求，并把上游响应转换回 Responses 格式。

适用场景：

- 上游只提供 OpenAI Chat Completions 兼容接口。
- 需要 function tool 映射、`reasoning_effort` 映射或 `web_search` 透传控制。

### responses

```yaml
outbound: responses
base_url: https://responses-gateway.example.com/v1
# 默认 path: /responses
```

上游本身是 OpenAI Responses 协议时，代理进入全透传模式：

- 请求体和响应体不做协议转换。
- 仅按 provider 配置注入 `Authorization: Bearer <api_key>`。
- 如果 `models[].mapped_model` 与入站模型名不同，会改写请求体中的 `model` 字段。
- `/models` 仍按本地配置暴露上下文窗口。

透传模式下，工具语义完全交给上游处理。

## models 写法

`models` 支持字符串或对象两种写法。

```yaml
models:
  - deepseek-v4-flash
  - name: gpt-5.5
    mapped_model: deepseek-v4-pro
    context_window: 1000000
    max_context_window: 1000000
    auto_compact_token_limit: 900000
    effective_context_window_percent: 95
    display_name: DeepSeek V4 Pro
    description: DeepSeek coding model
```

| 字段 | 说明 |
|------|------|
| `name` | 入站模型名，作为路由键，也是 `/models` 暴露的 slug |
| `mapped_model` | 当前 provider 的出站模型名，省略时默认与 `name` 相同 |
| `context_window` | 暴露给 `/models` 的上下文窗口 |
| `max_context_window` | Codex 配置覆盖时的上限，省略时与 `context_window` 一致 |
| `auto_compact_token_limit` | Codex 自动压缩阈值 |
| `effective_context_window_percent` | 可用上下文百分比 |
| `display_name` | 展示名 |
| `description` | 描述 |
| `base_instructions` | 覆盖该模型的 Codex 系统提示词 |

字符串写法表示入站/出站模型同名，只参与路由，不会单独声明上下文窗口。若希望模型出现在 `/models` 中，需要配置模型级 `context_window`，或设置全局 `defaults.context_window`。

## defaults

```yaml
defaults:
  context_window: 131072
  effective_context_window_percent: 95
  base_instructions: ""
  model_template: /path/to/model_info.json
```

| 字段 | 说明 |
|------|------|
| `context_window` | 未在模型对象中配置上下文窗口时的全局回退值 |
| `effective_context_window_percent` | `/models` 的全局可用上下文百分比，默认 `95` |
| `base_instructions` | 全局覆盖 `/models` 条目的系统提示词 |
| `model_template` | 自定义 Codex `ModelInfo` 模板 JSON 文件 |

## options

```yaml
options:
  passthrough_unknown: true
```

`passthrough_unknown` 控制 Chat 出站时是否透传未识别的顶层字段。provider 级同名字段优先于全局配置。

## 路由规则

请求进入代理后，路由流程如下：

1. 读取入站请求体中的 `model`。
2. 在启用的 provider 中查找 `models[].name` 声明该入站模型的候选。
3. 候选确定后，使用该 provider 对应 model 条目的 `mapped_model` 得到出站模型名。
4. 候选只有一个时直接使用。
5. 候选有多个时按 `weight` 加权随机选择。
6. 未命中时，如果存在单 provider 兜底配置，则使用兜底 provider。
7. 仍未命中时返回 `model_not_found` 错误。

## 安全建议

- 优先使用 `api_key_env`，避免把真实 Key 写入 `config.yaml`。
- 保持 `config.yaml` 与 `.env` 不提交到版本控制。
- 本地开发建议监听 `127.0.0.1`。
- 如果必须暴露到局域网或公网，请在外层增加认证、访问控制和日志审计。
